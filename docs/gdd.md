# tro.gg — Game Design Document

The buildable spec for tro.gg. If you are an agent working on this codebase: this file is the source of truth for game rules, data, naming, and scope, together with [analytics.md](analytics.md) (events and flags) and [world.md](world.md) (setting, tone, and naming for UI copy). [challenge.md](challenge.md) is human-facing background; you rarely need it. When code and this document disagree, flag the conflict — don't silently pick one.

## How to use this document

- **Glossary names are canonical.** Use them in code, schema, events, and UI. Don't invent synonyms (it's a `node`, not a "resource spawner").
- **Constants marked *(initial)* are starting values.** Keep them centralized in shared constants; make them remotely configurable only when runtime tuning or experiments are useful.
- **The invariants section is non-negotiable.** If a task seems to require breaking one, stop and ask.
- **Roadmap notes are planning context, not gates.** Do not block a task just because it used to belong to a later phase.
- **Custom analytics events and feature flags are registered in [analytics.md](analytics.md)** in the same change that introduces or changes them.

## Glossary

| Term | Meaning |
| ---- | ------- |
| trogg | A player character. Lowercase in prose, `player` in code/schema. |
| the dark | The hostile expanse beyond every hearth's light. Also used informally for unclaimed ground ("out in the dark"). |
| spark | The ember of the tribe's fire that animates every trogg. Always present; its brightness is a trogg's presence state, not a resource it holds or spends. |
| bright | A trogg's state while its player is connected and playing: full judgment, full combat, full access to the frontline and the dark beyond it. |
| ember | A trogg's state after its player disconnects, while `kindlingCharge` remains: it stays in the world, working safe interior ground on instinct, feeding the stockpile at reduced efficiency. `ember` is derived state (see Data model), not stored. |
| dormant | An ember trogg whose kindling charge has run out: settled at the nearest hearth, present and visible, no longer contributing until its player returns. |
| kindling charge | Earned budget of ember-time, accrued by bright play and spent while ember; decays if never renewed. Ties a trogg's usefulness while absent to its player actually showing up. `kindlingCharge` in code/schema. |
| the stockpile | The tribe's one shared pool of raw resources. Every trogg — bright or ember — deposits into it directly; nobody withdraws from it or owns a share. `stockpile` in code/schema. |
| hearth | A fire that shelters the ground around it: dark creatures cannot cross into its light. |
| the Hearth | The origin region of the world zone (see "region") — the tribe's town, where the First Fire burns and every newborn first emerges. Not a separate zone; ordinary ground in `world`, just the one hard-coded region rather than a procedurally placed one. |
| the First Fire | The one hearth that never gutters — the tribe's origin, at the Hearth, always lit. |
| brazier | A hearth lit at the frontline to hold and claim an entire region. `brazier` in code/schema. |
| the frontline | The boundary between the tribe's claimed regions and the penumbra beyond them. |
| region | One of an unbounded number of named areas of the world, each resolved on demand from a deterministic capital lattice (`shared/worldgen.ts`) rather than a fixed list — a colour/decoration character and a name for part of the map, and the unit the tribe claims: **interior** (claimed), **penumbra** (adjacent to a claimed region, not yet claimed), or **unreached** (neither). `regionAt` resolves a tile to its region. |
| claim | To clear every living dark creature out of a penumbra region and set a brazier down there for free, turning it interior (see "The fire and the dark" → Territory claiming). |
| penumbra | Every region adjacent to a claimed region but not itself claimed — scoutable, dangerous, governed by no hearth. |
| dark creature | A hostile inhabitant of the dark and the penumbra: aggressive on sight, kept off any lit ground by the hearth rule. Specific species are open (see Roadmap). `dark_creature` in code/schema. |
| zone | One contiguous area of the world. The unit of subscription, rendering, and chat. |
| tile | The grid unit. Positions are integer tile coordinates within a zone. |
| node | A gatherable world object (rock, mushroom). Has a type, a state, and a respawn timer. |
| boulder | A mineable rock. Sits on an unwalkable tile; a pickaxe mines it into Stone. `boulder` in code/schema. |
| tree | A choppable tree. Sits on an unwalkable tile like a boulder; an axe fells it into Wood. `tree` in code/schema. |
| interact | The generic action key (`E`). It picks up ground items and picks up / puts down tile-sized carryables; pickup scans adjacent tiles and prioritises the tile the trogg faces, while put-down uses the faced tile first. Future effects (switch, fire) hang off the same key. |
| carry | A trogg holding a tile-sized object on its person — the object leaves its tile until put down. No carryable exists at this design's outset; the mechanism is generic infrastructure for whatever hangs off it later. `carrying` in code/schema. |
| action | A timed activity a player starts on a node (mine, forage). One action at a time per player. |
| skill | A progression track (mining, foraging, combat). XP and levels per skill, per player, earned only by bright play. |
| recipe | A crafting definition: inputs → output, skill requirement. |
| guest | An anonymous player (self-issued anonymous session). Has a generated name until upgrade. |

## Design pillars

1. One mechanic shipped at a time. The game is always playable.
2. Players are residents, not spectators. Guests join in seconds, no signup wall.
3. Instrumentation is first-class — the dashboards are part of the project, built in the open. Experiments are run transparently: players are told when they're in one.
4. One shared world. No instancing below ~1,000 concurrent in a zone; scaling problems are solved when the graphs demand it, not in advance.
5. Programmer low-poly 3D for now; visual polish and art direction are deferred.
6. The fire is fed by presence, not population. A trogg's usefulness while its player is away is earned by real, attended play and decays without it — so the tribe's strength always reflects who actually shows up, not how many accounts exist.
7. Active play expands; ember play only sustains. Nothing new is ever discovered, fought for, or claimed by a trogg whose player isn't there — an idle account can keep the fire fed, never push it forward.

## Core loop

Fire-and-dark grammar: gather to feed the fire → the fire holds the dark back → push the fire outward → repeat, at whatever pace the tribe is actually present for.

1. Land on tro.gg → exist in the world within seconds as a guest (no signup wall), waking as a fresh spark in the Old Dark.
2. Mine your way out of the birth cave and emerge at the Hearth, built around the First Fire — the one flame that never goes out.
3. While bright: gather (mining, foraging — feeds the stockpile directly, no personal hoarding), fight and scout beyond the frontline for rare finds, help clear a penumbra region of every dark creature in it so the tribe can claim it, train skills, craft and upgrade your own gear.
4. Log off, and your trogg doesn't vanish: on its earned kindling charge it goes ember, keeps working safe interior ground on instinct, and quietly adds to the stockpile until the charge runs out and it settles dormant by a hearth.
5. The stockpile pays every brazier's upkeep — clearing and claiming new ground itself costs nothing else. Starved, the deepest claimed region's brazier gutters first and the dark reclaims that ground — never past the First Fire, and never undoing what's already been explored.
6. Chat renders as speech bubbles over heads (and in a side panel).

## Systems

### Movement

Two input modes, both supported:

- **Click-to-move:** click a tile → the trogg walks a pathfound route to it. Click a node or obstacle → route to the nearest reachable tile beside it (get as close as possible), then act.
- **WASD / arrow keys:** hold to move, release to stop. **Free camera-relative movement:** W walks where the camera looks, S backs up, A/D strafe, and held keys combine (W+D walks the forward-right diagonal at the same overall speed — the shared projection normalises velocity). A press moves (and faces) immediately — no turn-in-place beat — and a trogg can rest anywhere, not just on tile centres. Turning the camera mid-walk curves the walk with it: the client re-maps the held intent when the camera heading crosses a 15° bucket (still input-driven — a drag is input; never per-frame resends of an unchanged heading). On the wire a heading is an integer vector with axes in ±`DIR_SCALE` (1000); the server clamps the axes and the projection normalises, so magnitude never buys speed and nothing is trusted (invariant 3).
- **Sliding collision:** moving into a wall clamps flush; moving diagonally along one slides — the blocked axis stops, the free axis keeps going — so free movement never wedges on geometry. Collision is continuous (the footprint never enters an unwalkable tile), derived identically on every client and the server from the one shared projection.
- **Facing stays cardinal:** the synced standing facing (`player.faceX`/`faceY`) is the heading's dominant axis, so the tile mechanics that hang off facing — interact, attack, throw — keep their crisp grid targeting on top of free movement. `facingTile` treats a *mostly-cardinal* press (minor axis ≤ 0.35 of the major) as facing its dominant neighbour, so a slightly rotated camera doesn't break faced-tile targeting.

Both are **input-driven, not per-frame.** The client writes a movement or facing intent only on input transitions — a click (which sets a path), a key down/up, a direction change — never on a timer or every frame. Origins are **fractional**: an intent is sent from wherever the trogg stands, and the server settles the previous motion by deriving the same position from the same projection — nothing snaps, so there is no sub-tile distance to bank and a flurry of clicks or keypresses can't speed the trogg up (invariant 3). A held WASD run still re-bases its origin once per tile crossed (a motion transition, never a per-frame timer — invariant 2), so a trogg's position is only ever derived over the last tile rather than from a far-back run-start origin (see Obstacles for why). The server stores each player's motion as an origin `(x, y)`, a direction or a path, and `movedAt`; standing facing is stored separately as `faceX`/`faceY`. Position over time is derived from motion only, and clients render it locally between updates.

- Move speed: 4 tiles/sec *(initial)*, shared by both modes. A click overrides held keys and vice versa.
- **Running:** holding **shift** while moving runs at `RUN_SPEED_TILES_PER_SEC` (7 tiles/sec *(initial)*) instead of walking, controlled by the optional `running` flag (off → shift is ignored, movement stays at walk speed). Run state rides the synced motion intent (`player.running`), so `projectMotion` derives the same faster position on every client — no per-frame sync, no determinism mismatch (invariants 2 & 3). Releasing shift, stopping, or a change of direction re-bases the origin at the current speed (the `move` reducer settles before storing the new intent). Troggs show a faster, hunched run clip (the run gait in `src/game/rig.ts`).
- **Obstacles:** tiles carry a walkability flag; nodes and scenery (trees, rocks, walls) sit on unwalkable tiles. WASD clamps at the first unwalkable tile or the zone edge; click-to-move routes around them. Beyond the static tilemap, **dynamic obstacles** — boulders, trees, and dark creatures — block a trogg the same way a wall does (see [Boulders and trees](#boulders-and-trees) and [Dark creatures](#dark-creatures)). Walking into a dark creature, a trogg **stops flush** against it — its motion settles to a standstill on the tile, rather than holding a "still walking" intent — and resumes the instant the creature moves off. This matters because position is *derived* from the intent over elapsed time: a held walking intent jammed behind a moving obstacle banks travel it never made, so when the obstacle clears the derivation would fling the trogg to where the uninterrupted walk would have reached (and the server, re-deriving the same way, would snap it there). Settling to a standstill on contact keeps the intent honest, so there's nothing to release; a click-to-move route that a dark creature steps onto likewise stops the trogg flush rather than banking the rest of the path. The mirror hazard is a dark creature wandering onto a tile the trogg has *already* crossed: re-deriving from a far-back origin against the now-occupied tile would clamp the trogg flush against it — visibly rewinding it backward. A held WASD run re-bases its origin once per tile crossed (see Movement, above), so an already-passed tile sits behind the origin and can't reach back to rewind it (the WASD counterpart of the forward-only click-to-move projection and a wandering creature's tile-by-tile amble); any correction is bounded to the current tile.
- **Pathfinding:** the server runs grid `A*` over the zone's walkable tiles on each click, then **string-pulls** the route (`smoothPath`): from the fractional start it greedily keeps the furthest waypoint reachable in a straight line (line-of-sight tested with the same sliding-collision walker movement uses), so open floor collapses to one direct glide at any angle and only genuine corners keep bends. The smoothed `path` is stored on the player and clients animate along it — no client-side recompute, so no determinism mismatch to manage. A hop being traversed re-checks collision live, so a dark creature landing mid-glide clamps the trogg (the stall the client re-routes on) instead of being walked through.
- **No teleport-by-quit.** The stored `(x, y)` is the *origin* of the current move, never the destination. Position is only advanced by elapsed real time × speed along `path`, so clicking far away and quitting skips nothing — on return you're wherever the clock puts you (arrived only if enough time actually passed). A scheduled task may settle `(x, y)` to the path's end at `movedAt + traveltime`, but never before it.
- **Prediction (display only):** because clients sync *intents*, not positions, every client derives the same motion locally and animates other troggs continuously from their last known intent — no per-frame position sync. Clients map the server's `movedAt` onto their local monotonic clock so deployed latency does not make avatars trail their authoritative motion by a round trip. The local player's own accepted input is applied optimistically to display state at the (fractional) spot where it was sent; a matching server row is treated as an acknowledgement and does not restart the animation, while a mismatching row still snaps to server truth (invariant 3).

### Boulders and trees

Boulders and trees are the world's **gathering obstacles**: solid like walls (walkability is the static tilemap **minus** the tiles they occupy), removed only by the matching tool. There is no pushing and no picking them up — a boulder is a mining node, a tree is a woodcutting node, and clearing one is how a blocked way opens.

- **Nodes have health and take swings** (`BOULDER_MAX_HEALTH` 45, `TREE_MAX_HEALTH` 54 *(initial)* — roughly three average hits each). Every tool swing that reaches its node rolls the tool's `WEAPON_DAMAGE` range into the node's health, and the client answers each hit with the same feedback a creature gives: a white hit flash and a floating damage number. The breaking hit deletes the row and **drops the yield on the floor** by the node (`dropLoot` — the node's own tile is free once it breaks), whatever weapon dealt it. Nothing enters an inventory uninvited: picking the yield up is a conscious `E`, same as any loot.
- **Any weapon works, badly** (`OFF_TOOL_NODE_FACTOR`, 8% *(initial)*): when a swing finds no creature and no matching node, it scratches the nearest node for a fraction of the roll (never below 1) — a sword can whittle down the first tree for the wood that crafts a real axe, it's just a terrible saw. Creatures always outrank off-tool node scratches in the same swing.
- **Mine a boulder with a pickaxe.** Pressing `F` with a pickaxe equipped strikes the nearest boulder in melee reach along the aim (the same reach + swing-cone geometry as combat); the breaking hit drops one **Stone** where the boulder stood.
- **Fell a tree with an axe.** The exact mirror: `F` with an axe equipped chops the nearest tree in reach; the felling hit drops one **Wood** where the tree stood. The axe shares the pickaxe's overhead `chop` wield class.
- Both start from the zone's `boulders`/`trees` registry entries, seeded idempotently on first connect, and are healed by `healStaleWorld` after a map regen like every seeded row. Trees seed on dry open floor only (`assertZones` checks it).
- **Resetting:** the Commands panel can snap the player's current zone back to its registry boulder layout (the `resetBoulders` reducer clears and reseeds the zone). Behind the `boulder-reset` flag — off, the panel hides the boulder reset action. The same panel resets dark creatures (see "Dark creatures", below).

### Interacting (pick up and carry)

`E` is the generic **interact** key. It picks objects up from adjacent tiles and uses the faced tile first when several candidates are in reach, so later interactions (flip a switch, light a fire) can hang off the same key without adding a new control.

- **Pick up / put down is a toggle for tile-sized carryables.** Empty-handed, pressing `E` beside one **lifts it onto the trogg**: if several candidates are adjacent, the one on the tile the trogg faces wins. It leaves its tile and rides on your person, drawn as a full-size held overlay above the head (the same held-item layering as [Avatars and equipment](#avatars-and-equipment)); pickup changes its position, not its scale. Pressing `E` again **puts it down** on the faced tile (or the nearest free tile), re-materialising it in the world. A trogg carries at most one thing. No carryable exists at this design's outset (see "carry" in the glossary); this is the generic hook a future one hangs off. Boulders are **not** liftable — they're mining nodes (see [Boulders and trees](#boulders-and-trees)).
- **Throw what you carry.** Pressing `F` while carrying throws it in the faced cardinal direction as an instant, tile-based action. It travels up to `THROWN_OBJECT_RANGE` (4 tiles *(initial)*), stops at the first wall or solid object, and damages the first living trogg or dark creature it hits by `THROWN_OBJECT_DAMAGE` (40 *(initial)*). The thrown object lands at the farthest clear tile, or near the target it hit.
- **Ground items are picked up by radius, not facing** (`ITEM_PICKUP_RADIUS`, 1.75 tiles *(initial)*, centre to centre): empty-handed `E` lifts the **nearest** ground item within reach into inventory — a drop at your feet is always liftable regardless of which way you look. Every ground item carries a **pickup sparkle** (gold motes circling above it), the universal "you can E this" signal — loot, equipment, and player drops alike. Starter pickups at the Hearth are a pickaxe, shovel, axe, sword, shield, and torch. Items are not solid.
- **Carried things leave the world.** A carried entity stops existing as a world row while held and is re-inserted on drop, wherever the trogg is standing when it puts it down.
- **No tick, server-authoritative** (invariants 1 & 3). `interact` is an input-driven reducer. The client passes its current heading; the server re-derives the trogg's tile and acts only on adjacent entities or ground items, preferring the faced tile when there are multiple candidates, so the client can't reach past its neighbours. Carrying changes nothing per-frame — the held thing simply moves with its carrier, whose position is already derived.
- **Nothing is orphaned.** On disconnect the trogg drops what it holds where it settles, unless it's recalled to a hearth first (see "The fire and the dark" → Presence); the carried kind is durable on the player row, so even a mid-carry restart loses nothing.
- Behind the optional `interact` flag (off → `E` does nothing).

### Dark creatures

Hostile inhabitants of the dark and the penumbra beyond the frontline — the same wander/motion/collision engine that once drove ambient roaming Hogs now driving something that wants troggs dead. This is presence and threat, not narrative: no dialogue, no trade, no taming (yet).

- **Aggressive on sight.** A dark creature that spots a bright trogg within `DARK_CREATURE_AGGRO_RANGE` *(initial)* breaks its wander and closes the distance, then attacks on the same swing/hit-circle grammar as a trogg (see Combat). There is no stealth or line-of-sight model yet — sighting is range-based, like earshot.
- **Bound by the light** — the central rule: a dark creature cannot enter any tile of a region whose brazier is lit (see "The fire and the dark" → Territory and permanence). This is solid ground it can't cross, the same way a wall blocks a trogg — the whole region, not a ring around the fire. A chase that reaches a claimed region's edge stops there; the creature paces the boundary rather than crossing it. This is what makes the frontline legible at a glance and what keeps the interior — and every ember trogg grazing it — categorically safe without a single invulnerability flag.
- **Same motion model as troggs.** A dark creature carries an intent — origin `(x, y)`, a direction, `movedAt` — and clients derive its position with `projectMotion`, so it's never per-frame synced (invariant 2). Chasing swaps the wander tick's random turn for a turn-toward-target — still one write per state change, not a tick.
- **Territory-linked respawn** (full rule under "The fire and the dark" → Territory and permanence): a dark creature killed on ground the dark still owns respawns — the dark replenishes what's its own. A dark creature killed on lit, claimed ground stays dead — nothing dark returns where the tribe holds the light.
- **Server-owned, no identity**, like the roaming Hogs they replace: no player-facing name or persistence beyond the row, seeded and wandered by the same scheduled-reducer pattern (re-arming only while a player is online, so an empty world produces no diffs and no work — invariant 1).
- **Species are open.** The first bestiary entries, their names, stats, and any Umbra-touched (harsher, frontline-warped) variants of the same base models are unspecced — see Roadmap. Whatever ships reuses the jointed rig (Avatars and equipment) and the crowd/instancing/budget machinery Hogs proved out, so a new species is a model builder and a stat row, not a new engine.
- **Loot.** A killed dark creature drops its loot — species-specific, scaled by size — as ground items near the corpse, the same drop/decay/cap mechanics as before.
- **Resetting:** the Commands panel can snap the player's current zone dark creatures back to their registry population, the mirror of `resetBoulders`. Behind a flag, like every other Commands panel action.

### Camera and rendering

- 3/4 top-down (RuneScape-2004 / Stardew view), now rendered as a real 3D scene: a fixed camera above and south of the zone, tilted down at the cave.
- Rendered with **Three.js** (WebGL) on a Vite + TypeScript client (`src/game/`). The world is low-poly flat-shaded 3D built entirely in code — no modelled assets, no editor files. Terrain reuses the procedurally painted cave textures (nearest-filtered so the floor stays chunky); walls are beveled blocks read from the same tilemap movement collides against; glowmoss tiles cast real teal point lights; a warm key light casts shadows and dark fog closes the cave in. The camera auto-fits the whole zone at the 3/4 angle on resize; click-to-move raycasts the pointer onto the floor plane to pick the tile. The client subscribes to the zone's SpacetimeDB tables and draws them; all authority stays server-side (invariant 3).
- **Draw budget.** The render loop caps at `FRAME_CAP_FPS` (60 *(initial)*) — ProMotion displays would otherwise run 120Hz for double the heat and no gameplay benefit. A zone's trees and boulders draw as a handful of whole-zone instanced meshes (`src/game/nodes.ts`, one draw per template part; the hit flash rides per-instance colour), only the nearest `CREATURE_BUDGET` creatures of each kind render as full jointed rigs, and identical shapes and immutable flat colours across every code-built model share pooled GPU geometries/materials (`src/game/pool.ts`; anything a model mutates per instance — player tint, hit flash, downed fade — stays per-instance).
- **The far crowd.** Creatures beyond the full-rig budget don't vanish — they render as species-shaped, species-coloured instanced silhouettes (`src/game/crowd.ts`, two draw calls for the whole distant population) that still glide and turn with their live projected motion, so a zoomed-out world reads inhabited. Hidden-tier creatures keep projecting, colliding, and making sound either way; corpses get no silhouette.
- Visible in-game HUD surfaces — chat history/input, the top-left icon toggle bar (Appearance `P`, Inventory `I`), and the **game menu** (`Esc`) — are HTML/CSS overlays above the canvas; the pre-alpha **Commands drawer** (`` ` ``) is deliberately on the opposite side — a lone toggle in the top-right corner opening a full-height right-edge drawer — so debug chrome never crowds player controls, so the browser owns layout, text input, focus, IME, and resize. They use `pointer-events` so a click on a panel is consumed by the DOM and a click on open space falls through to the canvas (click-to-move). The toggle bar is an accordion: opening one menu closes the others. **`Esc` is the single menu key** (`src/ui/menu.ts`): with something open it closes it (a drawer, the map, or the menu itself); on a clear screen it opens the game menu — a pause-style modal whose root is a short list (**Settings**, **Help**, and **Log out** when signed in); picking Settings or Help drills into that page with a Back control, and Escape steps back to the list before it closes. (The guest **Claim account with Discord** button lives under Appearance.) A focused chat/rename field owns `Esc` (blur), so it only reaches the menu when you aren't typing. World-space labels, health bars, and speech bubbles over troggs are camera-facing canvas sprites inside the 3D scene (`src/game/overlays.ts`), drawn depth-test-off so a wall never swallows a nameplate. The Help tab lists only enabled controls, and the Commands drawer exposes only enabled debug tools.
- **First-time onboarding (the coach, `src/ui/coach.ts`).** New players can't discover the verbs by clicking — pickup is `E`, equip is a right-click in the pack — so the first time a trogg reaches a milestone it raises one short hint card (bottom-centre, above chat), shown **once ever** (seen ids persist in `localStorage`; a returning player is never re-taught). Milestones announce via `coachHit(id)` (a window event, so no component holds a coach reference): waking in the birth cave (`find-pickaxe`), the first inventory gain (`first-pickup`), the first equip (`first-equip`), and first stone/wood (`mined-stone`/`chopped-wood`). Hints queue and show one at a time; click or wait to dismiss.

### Audio

- Sound effects are client-side feedback only; they never affect reducers, synced state, or analytics.
- The mix keeps footsteps under the theme: self and nearby strides sit low; larger creatures **thud** (slow, pitched down) where smaller ones patter.
- **The game theme is composed in code** (`GameTheme` in `src/theme.ts`), like every other asset: a slow generative ambient — two detuned pads breathing a low drone under sparse pentatonic plucks with a feedback-delay tail — via WebAudio, started on the first user gesture. No music files, no licensing.
- **World sounds are distance-based** (`EARSHOT`, 14 tiles *(initial)*): nearby troggs' footsteps, other creatures' patter (the same strides pitched by size), and creature calls play with linear falloff from the local trogg and are silent beyond earshot, as are boulder moves heard from afar. A world of a hundred creatures never becomes a hundred footstep tracks.
- The current audio surface is event-driven: local trogg footsteps fire when movement starts and when the trogg crosses tiles; a moving boulder row (a throw landing) plays a settle hit and a felled tree plays the same crash; a creature's heading change may play a sparse, low-volume, distance-faded call after the initial subscription snapshot; chat, commands, command errors, and the ghost haunt play short UI/ghost cues; a live inventory gain plays a pickup cue voiced by material (stone rattles, wood knocks, anything else the soft item thump).
- **The player owns the mix.** Every sound in the game belongs to exactly one of four categories — **Music** (the theme), **Footsteps** (troggs and other creatures), **World** (boulders, creature calls, the ghost), **Interface** (chat, commands, pickups, errors) — mapped per cue in `src/audio.ts` (`CUE_CATEGORY`) and enforced by the type of `src/sound-settings.ts`, so a new cue must pick a slider. The Settings tab of the game menu (`Esc`) holds a 0–100% slider per category; levels apply live (the theme re-mixes without restarting) and persist in `localStorage`, which the landing page theme honours too.
- Browsers can block playback before the first user gesture, so rejected play attempts are ignored and retried on the next cue.
- Source files and license notes live under `assets/audio/ATTRIBUTION.md`. Do not wire entire packs wholesale; pick short cues, keep volume conservative, and trim/normalize before promoting a candidate into `src/audio.ts`.

### Avatars and equipment

- A trogg (and every other creature) is a **jointed 3D model**: a low-poly flat-shaded body built in code on a shared skeleton, animated by real animation clips. Troggs and other creatures share the joint vocabulary, so equipment renders the same on any of them.
- **Creature rig (skeleton + clips).** Every creature is a hierarchy of named joint nodes — `Bob` (the gait dip), `LegL`/`LegR`, `Torso`, `ArmL`/`ArmR`, `Head`, plus the equip attach nodes `HandR` (main hand) and `HandL` (off hand) — built per species in `src/game/creatures.ts` over the shared machinery in `rig.ts`. Animation is **data played by the engine**: idle (breathing), walk, run (faster stride, forward hunch), and one attack per **wield class** (wind-up → strike → recovery, timed to the synced equipment-use impulse) are procedurally authored `AnimationClip`s played through Three's `AnimationMixer` with crossfades. Weapons declare their wield class on the item registry (`ITEMS[..].wield`): the sword **stabs** (a level thrust), the pickaxe **chops** (overhead slam), the shovel **scoops** (low dig and heave), and bare fists (or anything unclassed) **swing**. Each clip drives the arm, torso, *and the hand node as a wrist* that aims the held item's business end through the arc — authored once per class in `rig.ts`, parameterized by the species' `GaitSpec`, so every creature on the joint vocabulary performs every weapon correctly. Gaits are layered — a legs layer (legs + bob) under an upper-body layer (arms + torso) — and an attack replaces only the upper layer, so attacking on the move never stops the feet. There are no per-facing frames — the model yaws smoothly toward its heading; the cardinal `Facing` survives as game logic (movement intents, interact targeting). Equipment use plays the attack clip, so the creature's **arm actually swings**; a held item is parented to the hand node, so it rides the swinging arm with no placement math. Taking damage plays a **hit flinch** — a short recoil opposite the facing plus a solid white flash (materials driven by `flash`), triggered when a creature's health drops; every creature flinches the same way.
- **Held items** (pickaxe, shovel, axe, sword, shield, torch) render as models parented to the rig's hand nodes — a **main hand** (`HandR`) and an **off hand** (`HandL`), so combinations like sword + shield or sword + torch work. Live held-item behavior — a light, animated flame cels, a holding-arm pose — is authored **once per item** as its `HeldFx` (`src/game/items.ts`), consumed verbatim by both the game and the `/preview` page, so a new weapon never redefines how it is held per surface. An equipped **torch** carries a pool of warm firelight — the landing page's torch made portable: hard-swapped sculpted flame cels, an off-hand arm posed aloft after each mixer update, and a point light at the flame flickering on offset sines; torch lights have their own budget — only the nearest `TORCH_LIGHT_BUDGET` visible torch-bearers shed real light, the glowmoss budget's reasoning — so a torchlit crowd never multiplies the forward renderer's per-fragment light cost. A new holdable is one small model builder in `src/game/items.ts` (grip at the origin, business end along +y), not a new character: parenting to the hand makes it swing through the walk/run cycle and lead the strike on attack for every species at once. Occlusion is free — the depth buffer handles it. Items rest **perpendicular to the forearm** — the way a fist grips a haft — tuned by a per-item rest pitch (sword up at the ready, pickaxe hafted forward, shovel low); the shield alone straps along the forearm, face out, raised into guard. Each species declares an item **fit scale** on its model (`finishCreature`), so each species grips gear at its own scale — the minimal form of the per-(item, species) fit in the layered-equipment design below. The inventory, equipped-slot, and spawn-button icons render from these same models (`src/game/icons.ts`), so every surface shows the exact object the world renders; the HUD panel toggles (inventory/commands/appearance/help) render from tiny low-poly props through the same rig, so no icon in the HUD is flat art.
- **Armor (later)** attaches the same way to body joints (head, torso). The joint vocabulary reserves the anchors now; armor models are added with the mechanic.
- What's equipped rides the zone's player sync, so others see what you're holding. Pressing `F` uses what the trogg is currently holding or wielding without stopping movement — and holding `F` keeps using it at the swing cadence, so chained attacks are a hold, not a tap per swing: a carried object is thrown, a pickaxe mines the nearest boulder in reach of the swing into Stone, a sword attacks the nearest creature in reach (GDD [Combat](#combat)), and a shovel currently shows its synced use animation only. The tools, the stone resource, and the boulder are low-poly props in the same code-built style as the creatures (`src/game/items.ts`); the world palette lives in `src/game/palette.ts`.
- **Model avatars:** a trogg renders as its jointed 3D model (styles `moss`/`stone`/`ridge` vary palette and brow), feet anchored at the centre of its tile. The per-trogg colour multiplies the body's materials, so the same trogg is the same colour for everyone, every session; your own trogg gets a ground ring so you can pick it out. (The retired `avatar-sprites` flag is noted in [analytics.md](analytics.md).) The dev `/preview` page inspects every creature model, clip, and held item, URL-addressable for shareable deep links; concept references live in `docs/art-refs/`.
- **Two appearance axes — style and colour.** A trogg's look is a **body style** (the model's shape and base palette) plus a **colour tint** over it; the two are independent, so any style can wear any colour. Each is the value the trogg chose, or — until it chooses — a stable default derived from its durable id (a deterministic projection, like a level from XP; `STYLE_UNSET` / `COLOR_UNSET` = -1 are the unchosen sentinels). Both ride the zone player sync, so the model (and the trogg's chat-name colour) update everywhere they're shown.
- **Trogg style (`trogg-restyle`):** styles come from a fixed list (`TROGG_STYLES` in `shared`: `moss`, `stone`, `ridge`) that vary the silhouette features (ear nubs / earless crag / horns) and base palette — same rig, different head and tone. A trogg picks one via the `restyle` reducer, which stores its chosen index on the `player` row (validated server-side, invariant 3). The optional flag controls whether the style buttons show in the Appearance panel.
- **Trogg colour (`trogg-recolor`):** the tint comes from a fixed palette (`TROGG_COLORS` in `shared`), chosen via the `recolor` reducer (the mirror of `restyle` on the colour axis). The optional flag controls whether the palette swatches show in the Appearance panel.
- **Ghost (`ghost-trogg`):** the cosmetic easter egg is its own bespoke model (`buildGhost` — a translucent draped sheet, two eye holes, stub feet), never tinted. Summoning it inserts a zone-scoped `ghost_haunt` row; live clients in that zone render the fresh insert once as a slow materialise, gentle drift, linger, and fade-out, while late joiners ignore the replayed snapshot.
- **Appearance panel.** Rename, recolour, and restyle are one top-left HUD icon toggle (`P`, beside Inventory) — everything about how your trogg *looks* in one place. Sign-in/log-out lives in the game menu (`Esc`), gated by `auth-enabled`.

#### Layered avatars and cross-species equipment (design)

The target the rig is being built toward: a creature is not one model but a **stack of attachments** — base body, eyes, hair, outfit, armour pieces (breastplate, helmet, …), and held items — parented to its joints. Cosmetics and equipment are attachments you add, so players look different and gear is visibly worn. This is the paper-doll / layered-equipment pattern, and it's cheap here: an attachment is a model parented to a named joint, it animates with the body for free, and the depth buffer does the occlusion.

- **The rig is a cross-species slot contract.** It generalises from today's hand nodes (`HandR`/`HandL`) to a fixed vocabulary of named **slots** — `head`, `eyes`, `face`, `chest`, `mainHand`, `offHand`, `back`, `feet` — each an attach node (a transform, plus an optional fit scale) defined **per species**, once — the pose comes from the skeleton, not per-facing tables. The slot *names and semantics are shared across species*, so any attachment that targets a slot can be placed on any creature that implements the rig. A new monster race = build its body on the joint vocabulary; it then supports every existing item.
- **Item identity is species-agnostic; item *rendering* resolves per (item, species).** An item (a breastplate) is one registry entry — id, slot, stats, drop, equippable — independent of who wears it. How it *looks worn* is resolved per wearer: a **bespoke per-species model variant** when authored, else the shared model scaled by that species' slot fit, so a new species wears every existing item on day one (polished with bespoke models later). Stats travel with the item; fit travels with the wearer.
- **Runtime attachment, synced loadouts.** The look of a creature is its equipped + cosmetic set. Each player syncs its slot choices (extending today's `equippedMainHand` and the appearance style/colour axes); every client parents the loadout's models to the wearer's joints — no texture baking, no combination-space caching. Per-attachment material tint is how colour variety (hair, armour) rides without separate models per colour.
- **Cross-species lifecycle.** A monster dies → drops the item as a species-agnostic `ground_item` → any creature picks it up and equips it → it renders with that wearer's own fit. The drop, the inventory row, and the stats are the item's; the silhouette is the wearer's.

Current state this generalises from: **both hands are equippable and synced gameplay-side** — `equippedMainHand` and `equippedOffHand` (each with its inventory-row pointer) route by the item's slot, a **shield** equips to the off hand from the inventory panel, and every client parents both item models to the wearer's hand nodes. The shield is the first off-hand item with a gameplay effect: it blocks `SHIELD_BLOCK_FRACTION` (30% *(initial)*) of every hit its wearer takes (see "Combat"); the torch, and any future off-hand item without a `block` stat, stays cosmetic. Every creature but nothing-special cases implements the full joint vocabulary, so any slot-targeted attachment can land on any of them. What's *not* wired yet is the wider slot vocabulary (head/chest/…), which lands with its first consumer. Attachments are then added cheapest-first (eyes/hair before armour).

Open questions to resolve before building: whether a slot carries a fit scale beyond the species-wide one shipped today, or a full transform; per-attachment tint vs separate models for colour variety; and how cosmetic-only attachments (hair, eyes) fold into the existing appearance/`restyle` model.

### Zones

- A zone has a slug, display name, integer width/height in tiles, and a tilemap: per-tile walkability plus scenery. Nodes and obstacles sit on unwalkable tiles.
- **Tile glyphs.** Each character of a zone's `tiles` rows is a tile glyph (`TILE_GLYPHS` in `shared`). `WALL_TILE` (`#`) is the only unwalkable glyph — `isWalkable` keys solely off `#`, so collision and pathfinding ignore the rest. The other glyphs are **cosmetic floor variants** that dress the stone so a zone reads as varied terrain instead of one flat fill: `.` plain floor, `,` gravel scree, `"` moss, `~` shallow water (walkable — a puddle the trogg wades through; an impassable pool would be a `#`-class glyph), `*` glowmoss. The client lays a tile-sized decal quad per variant (`src/game/terrain.ts`); they change rendering only, never movement (glowmoss also casts a cosmetic point light). `assertZones` rejects any glyph not in `TILE_GLYPHS`. Add a new variant by registering its glyph and an overlay painter together.
- **Zone definitions are a static code registry** (`ZONES` in `shared`, keyed by slug) — static design data like the item and node registries, not the durable `zones` table the data model lists. A zone's tilemap may be **procedurally generated**: `generateCaveZone` (`shared/worldgen.ts`) grows a cave from a fixed seed with pure deterministic arithmetic (mulberry32, no `Math.random`), so the client and the module derive the identical world — generation guarantees a solid rim, an open spawn plaza at the centre (`spawnAt`), full reachability from spawn (disconnected pockets are filled), and seeds (boulders, trees, starter tools, dark creatures) on open floor, all covered by tests. Change the seed or dimensions in the registry entry to reroll the world. The table is deferred until tilemaps need editable storage; until then a zone is a registry entry. A client subscribes to one zone's rows by slug (`WHERE zone_id = …`); zone dimensions and the tilemap (`tiles`, walkability read through `isWalkable`) come from the shared `ZONES` registry (imported by both the client and the module), so the grid is shared design data, not a per-session value. The client renders walls and floor variants from the same tilemap it collides against, so what's drawn is what blocks you.
- **Spawn and the hub:** every newborn digs out of its own private birth cave and emerges directly into **the Hearth** — the origin region of the one seamless `world` zone, built around the First Fire (see "Onboarding: the Warren" and "The fire and the dark"). There's no separate hub zone or checkpoint to cross: the Hearth is ordinary ground in `world`, reached by walking out of the cave, not by an instance transfer or a progression gate.
- **The world is ONE seamless zone** (slug `world`): a landmass with no edge, grown like terrain rather than assembled from blocks — organic regions shaped by warped Voronoi around capitals resolved from an unbounded deterministic lattice (see "The fire and the dark" → Generation), and ONE cave automaton per region whose rock density comes from that region's biome, matched at the border so neighbouring regions blend with no seams. Impassability reads in the **z-axis**: walls and boulders rise, rivers sink — deep water is cut out of the floor and rendered as a recessed channel with dark banks, so terrain you can't cross never looks like floor. **Terrain placement rules** keep the world sensible: seeds, drops, and Commands-panel spawns land only on *dry* open floor (`isDryFloor` — never in rivers, fords, or pools), and roaming creatures keep to the banks (water blocks their wander like a boulder does). A trogg still wades through shallow water; **the Hearth** — the one hard-coded region, at the origin — is a real town at the spawn: a gravel plaza ringed by stone huts with doorways facing in, built around the First Fire. Regions are a colour/decoration character and a name for part of the map, not instances; `regionAt` resolves any tile to its region on demand. They're also the unit the tribe claims (see "The fire and the dark" → Generation): a region not yet interior or penumbra is impassable, regardless of what's actually drawn there.
- **The world heals under regeneration.** If the generator's formula ever changes (a tuning tweak, a bug fix), a previously-resolved region could redraw differently under a live database: `healStaleWorld` (on connect) detects any boulder/dark-creature/ground item on now-unwalkable ground and wipes + reseeds it, and a returning trogg stranded in new rock or water relocates to the **nearest safe dry tile** beside where it logged out (`nearestSafeTile`), falling back to spawn only when nothing nearby is open. Ordinary play never triggers this — a region's tiles don't move on their own once resolved.
- **The map is generated, not committed.** There's no single checked-in file enumerating every tile — `regionAt`, tile glyphs, and every per-region seed are pure functions of a tile's coordinates and the world seed (Generation, above), computed on demand and identically on client and server. This trades away hand-editing a specific tile or seed directly (there's no file left to touch — the generator's formula is the only lever) for a map with no edge to eventually run out of. Worldgen tests validate the generator's local invariants instead of one whole-map pass: a region's plaza is walkable and corridor-connected to its neighbours, seeds land on open floor, per-region density and capital spacing hold, and generation is deterministic.
- **The overworld map (`M`)** paints the world downsampled 2×2 and tinted per region palette (`src/ui/worldmap.ts`) — sparser than the world but derived from the same data, so it is always consistent with what's actually there. Every region paints its real terrain, fogged rather than blank past the frontier — penumbra lightly, unreached heavily — using the same two fog mixes the 3D view blends over its walls and floor. Unreached region names stay hidden, so the map never leaks what a scout hasn't found yet; penumbra names show, since you already know roughly what's there. It shows the trogg's own live position (never other players); `M`/`Esc`/click closes it, and it repaints on every open so a region claimed since it was last opened shows up without a reload.
- **The game opens in third person, behind the trogg** — a chase view at shoulder height, not top-down, so the world reads at its own scale from the first frame; the orbit camera takes over from there (drag to look around). There is no first-person mode. An orbit camera can only look *at* its target, so on its own it bottoms out at the trogg's shoulders: instead, dragging down past the floor parks the camera skimming the ground and turns the extra drag into an upward pitch (`src/game/controls.ts`) — keep dragging and you look up at the sky, the sun, and the peaks; drag back and it returns the same way. **The trogg is never eclipsed by terrain**: each frame the client marches the trogg→camera sight line through the shared rock-height model and, if rock crosses it, the render camera rises on its orbit sphere — same radius and azimuth, the smallest pitch lift that looks over the blocker — briskly eased up, gently eased down, with a short hold before easing down so corners grazing the line while pathing can't pump the camera. The player's zoom is never touched, and it's render-only: the orbit keeps the pose the player chose.
- **A shared day–night cycle** (12 min *(initial)*, wall-clock phased so every player sees the same time of day): the sun arcs east→west with the shadows tracking it, dusk fades the sky and haze, and night falls to a dim moonlit blue where the glowmoss carries the light. The dark presses harder at night — see "The fire and the dark".
- **Daylight and scale.** The continent lives under a **sun**: a sky backdrop with aerial haze, a bright warm directional sun (riding the camera focus with its shadow box) and a cool sky bounce — a world, not a cave. Rock height is **terrain-shaped**: a distance field from the walkable floor keeps rock beside paths at shoulder height and raises formation cores into **mountains** (up to ~9 tiles), while low-frequency noise makes ranges undulate and throws occasional summits — deterministic, so every client carves the same skyline. Low camera angles read as standing in a valley among peaks, and the sun rakes long shadows off them.
- **The client streams the world in proximity chunks** (`src/game/terrain.ts`): 32-tile chunks build nearest-first (throttled per frame) around the camera focus and dispose with hysteresis — Google-Maps style. Each chunk is one floor texture (tiles blitted from the biome patch painters, deep water included), one instanced wall mesh (per-instance biome colours), and glowmoss lights that spawn dark under a global budget (the nearest 8 within 48 tiles are live — a forward renderer pays every light on every fragment). Camera zoom is **capped** (`maxDistance` 34) with only a faint fixed depth haze for distance, unrelated to the frontier fog (Generation) — nothing nearby is artificially obscured; the M map is the wide view. Distant creatures/boulders/items hide beyond `CULL_RANGE`, and creatures additionally render under a **visibility budget** (the light budget's pattern): only the nearest `CREATURE_BUDGET` troggs — and separately dark creatures — draw at once, since each jointed rig is ~20 draw calls plus overlays and a shadow pass; hidden creatures still project, collide, and make sound, and their animation mixers pause. The shadowing key light rides the camera focus.
- Clients subscribe to players/nodes/chat **in their current zone only**. Within a zone, sync is deliberately naive (whole-zone queries) — see invariant 10.
- New zones ship incrementally; transitions are walk-to-edge or interact-with-passage. How later areas gate (checkpoints, light, communal projects) is an open thread.

### Chat

- Zone-scoped. Max 200 chars *(initial)*. Bubble displays 5s *(initial)*; side panel keeps recent history.
- Chat history renders as free-floating lines, each on its own translucent backing sized to the text — an empty chat shows nothing at all (no panel); the typing field is a real `<input>`; speech bubbles over troggs render as sprites in the 3D world. Chat content is added as a DOM text node, never as HTML markup, so a message can't inject markup.
- Server-side rate limit: 1 message/sec per player *(initial)*.
- Message content is **never** sent to analytics.
- **Pre-alpha Commands panel:** debug tools live in the top-left Commands panel — including a client-side **Hitboxes** toggle (`B`) that shows every creature's, boulder's, and tree's combat hit circle plus the local trogg's melee reach ring and swing cone (the cone follows the live aim and brightens while a use is in flight), not chat slash commands. The spawn grid uses icons rendered from the actual world models: one boulder button, one tree button, one button per dark-creature species/size, and one button per `SPAWNABLE_ITEM_IDS` item — the equipment (`pickaxe`, `shovel`, `axe`, `sword`, `shield`, `torch`) plus `stone` and `wood`, which are spawnable for material testing even though normal play gathers them through mining and woodcutting. Each click requests exactly one spawn; if someone wants ten of a creature, they press its button ten times. The `spawn` reducer places the thing on a nearby free tile around the caller, starting with the tile they face, and clamps inserts to `MAX_BOULDERS_PER_ZONE`, `MAX_TREES_PER_ZONE`, `MAX_DARK_CREATURES_PER_ZONE`, `MAX_GROUND_ITEMS_PER_ZONE`, and available floor so the UI can be abused without bypassing caps.
- **Ghost:** the Commands panel can summon the cosmetic ghost (a draped-sheet apparition; see [Avatars](#avatars-and-equipment)) at a server-picked random walkable tile in the caller's zone (behind `ghost-trogg`, fallback on). It fades in, drifts gently, lingers for a few seconds, and fades out. It inserts a capped `ghost_haunt` row so every live player in the map sees the same haunt; the row has no collision or durable gameplay effect, and late joiners ignore old rows.

### Identity

- Anonymous-first: SpacetimeDB issues each connection a cryptographic **Identity**; the browser stores the connection token it returns. Identity is the connection's own `ctx.sender` server-side, never client-asserted (invariant 3). Guests get a generated name `trogg-####` and exist within seconds, no signup.
- **Guest persistence:** the browser securely stores the SpacetimeDB connection token — not game state, which stays server-authoritative (invariant 3) — so a returning visitor reconnects with the same Identity and resumes the same trogg row with their progress intact. Clearing the browser or switching devices makes a guest a new trogg.
- **Surviving a redeploy:** the frontend and backend ship *independently* — the client to Cloudflare, the SpacetimeDB module to the VPS — so each deploy is handled on its own signal.
  - *Backend deploy:* SpacetimeDB is both the store and the live feed, so publishing a new module version closes every live socket at once. The client treats this as recoverable, not fatal: it shows a "reconnecting" overlay and probes for the server with exponential backoff + full jitter (jitter so the whole world doesn't stampede the instance the moment it restarts), then reloads once it's back. Because state is server-authoritative and re-derived from subscriptions, and the stored token resumes the same trogg, this loses nothing — players stay in the world across a deploy instead of being silently logged out. The reload also pulls the latest frontend as a side effect. Emits `connection_lost` (analytics.md).
  - *Frontend deploy:* a client-only deploy leaves the socket untouched, so there's no disconnect to react to. Each build is stamped (`__BUILD_ID__` + a shipped `version.json`); the client polls that stamp (every 60s and on tab focus) and, when a newer build is live, shows a dismissible "new version — refresh" prompt rather than forcing a reload — the old client keeps working against the unchanged backend, so the player refreshes when it suits them. Emits `client_update_available` (analytics.md).
- **Signing in** upgrades a guest to an account via **SpacetimeAuth** (SpacetimeDB's managed OIDC provider; Discord is the enabled login). SpacetimeDB derives a *stable* Identity from the OIDC token's `iss`+`sub`, so the account — not the browser — now anchors the synced state and the trogg resumes on any device. The browser runs the OIDC Authorization-Code-**+-PKCE** flow (a public client: no client secret in the bundle, invariant 8); the module trusts only the SpacetimeAuth issuer as an account provider (invariant 3). Account creation and the upgrade fire `player_named` alongside `posthog.identify()`, merging the guest's history.
- **Multiple live sessions for one account:** two tabs or devices signed into the same account share the same `player` row and therefore the same trogg. Any live session can send input; the other sessions simply observe the shared motion intent and render the trogg following it. Presence is tracked per socket in the private `player_connection` table, so opening a second session does not restart the trogg's movement, and closing one session does not mark the shared trogg offline while another session remains connected.
- **Claiming** (folding a guest's trogg into the account, since the two are different Identities): the guest's browser mints a one-time nonce, registers it under the guest Identity via `startClaim`, then signs in and redeems it as the account via `redeemClaim` — both sides proven, never a client-asserted identity (invariant 3). The guest's chosen name carries over (a generated `trogg-####` never overwrites a name the account already chose); the guest row is then absorbed. A fresh device with no guest just signs in and resumes the account directly. Nonces expire after `CLAIM_CODE_TTL_MS`.
- **Changing your name:** the `rename` reducer swaps the generated `trogg-####` for a chosen one, validated server-side. Names: unique, 3–20 chars, alphanumeric + hyphen. The new name takes effect everywhere it's shown — the nameplate over the trogg and the denormalised `name` on the player's past `chat_message` rows are both rewritten, so nothing keeps showing the old name.
- The account action lives in the game menu (`Esc`) — Sign in with Discord for a guest, Log out when signed in (log out drops back to the landing page). Rename, colour, and style live in the Appearance panel. Sign-in uses the browser OIDC redirect.

### Skills and XP

- Skills exist per player, per skill, as accumulated XP. Level is derived, never stored.
- **XP is earned only by bright play.** An action completed while connected and actively playing grants XP as before; the same action completed by an ember trogg on instinct (see "The fire and the dark" → Presence) deposits into the stockpile but grants none. Ember troggs work; they don't grow. This is the mechanism behind design pillar 7 — active play is the only thing that ever gets better.
- Total XP to reach level L: `50 × (L − 1)²` *(initial)* — level 2 at 50 XP, level 10 at 4,050.
- Level cap: 50 *(initial)*.
- First skills: **mining**, **foraging**, and **combat** — clearing a region and pushing the frontline out are no longer a future event to spec around; they're the game, so combat XP ships with them rather than waiting.

### Gathering (nodes and actions)

- Clicking a node routes the trogg to the nearest reachable tile adjacent to it (nodes are unwalkable). On arrival the action starts: the server validates (adjacent, node available, no action in progress), writes the action with an end time, and schedules its completion server-side (`endsAt` persisted so it survives a restart).
- **Completion deposits straight into the stockpile.** A bright trogg's completed action grants XP to the player and adds the yielded item to the shared `stockpile` (see "The fire and the dark" → The stockpile) — never to the player's own inventory. Nothing is gathered into a personal stash; the tribe's resources are the tribe's from the moment they leave the node. The node flips to `depleted` and schedules its respawn. If the player walked away mid-action, the action cancels.
- **Ember gathering is the same deposit, at a fraction of the rate and with no XP** — see "The fire and the dark" → Presence for the full ember-trogg mechanic.
- One action at a time per player. Starting a new one cancels the old.

### Inventory

- The top-left Inventory icon (`I`) opens the inventory/equipment panel. Starter pickup items are seeded from the zone registry as `ground_item` rows; pressing `E` while facing one moves it into the player's inventory and removes the ground row.
- **Personal inventory now holds only equipment and rare carried finds.** Bulk raw resources (Stone, Wood, Glowcap Mushroom) skip personal inventory entirely and deposit straight into the shared stockpile on gather (see "Gathering"). What's left in a player's own inventory is what's actually theirs: the pickaxe, shovel, axe, sword, shield, and torch they equip, plus anything picked up as a `ground_item` (starter tools, a dead trogg's dropped gear). A carried tile-sized object would ride outside inventory entirely, not inventoried — see "Interacting" — though none exists at this design's outset.
- Inventory has 20 visible carry slots *(initial)*. The HUD always shows all slot outlines, filled first by owned inventory rows and then empty slots, so players can see remaining capacity at a glance.
- Inventory rows store an item id and quantity, and each row occupies one visible carry slot. Equippable items (Pickaxe, Shovel, Axe, Sword, Shield) are non-stackable qty=1 rows, so two swords remain two visible slots and the equipped row is unambiguous. Stacking still exists in the item registry for a future stackable pickup, but neither current inventory category — equipment, rare finds — uses it today.
- Items are defined in a static registry (id, name, stackable, blurb). Holdable/wearable items also carry their slot and model. No item randomization.
- **Pickups announce themselves**: a live inventory gain (new row or a stack growing — never the boot sync) raises a bottom-centre toast with the item's icon and name; rapid gains of the same item bump one card's count instead of stacking duplicates (`src/ui/toasts.ts`).
- **Hovering (or focusing) any occupied tile — carry slots and both hands — shows a tooltip** with the item's name and registry blurb, so what a thing is for is discoverable in place. The same shared tooltip (`src/ui/tooltip.ts`) covers every HUD control (the top-left toggles, the Debug Tools toggle, the spawn tiles), always clamped inside the viewport. The page behaves like a game surface, not a document: text never highlights (real inputs excepted) and the browser's own context menu is suppressed — right-click belongs to the game.
- Equipping sets the slot the item belongs to — `equippedMainHand`/`equippedMainHandInventoryId` for tools and weapons, `equippedOffHand`/`equippedOffHandInventoryId` for shields — to an owned row (the item stays counted in inventory; equipment just references it). The reducer routes by the item's `slot`, so the two hands equip independently; equipping the row already in its slot toggles it off, and `0` clears both hands. See [Avatars and equipment](#avatars-and-equipment).
- **Right-click is the item verb**: a tile's context menu opens at the cursor — equip/unequip (equippables only, into the item's hand), **drop**, and **delete** (the entry arms into an in-place confirm; the second click destroys). Left-clicking a tile does nothing. Both drop and delete act on one unit — a stack decrements, a qty=1 row is removed; removing the equipped row unequips it. Drop lays that unit back into the world as a `ground_item` on the faced tile, the nearest free neighbour, or the trogg's own tile (the carried put-down placement), honouring `MAX_GROUND_ITEMS_PER_ZONE`; a full or boxed-in zone refuses the drop and keeps the item. Delete destroys the unit outright with no ground item, behind a confirm step in the panel.

### Combat

Pre-alpha combat is deliberately small and tile-based. Troggs and dark creatures are the current damageable characters; clearing a region's dark creatures (see "The fire and the dark" → Territory claiming) uses the same slow, stat-driven grammar rather than twitch mechanics.

- **Health:** each trogg has `PLAYER_MAX_HEALTH` (100 *(initial)*) health points. Health rides the `player` row, so everyone sees the same health bar and death state from server truth.
- **Dark-creature health:** a dark creature carries species-scaled health on its row. Damaged dark creatures show a small health bar. At zero health a dark creature becomes a **corpse**: settled where it fell, stopped, lying flat in the dead stance — not solid, not targetable, not liftable — until the regen sweep reaps it after `NPC_CORPSE_MS` (30s *(initial)*). Whether it respawns at all depends on whether the ground it died on is lit at the time (see "The fire and the dark" → Territory and permanence) rather than anything about the kill itself. The killing blow also **drops the creature's loot** — species-specific, scaled by size, rolled with the context RNG — as ground items on the nearest free tiles around the corpse, honouring the per-zone ground-item cap. Troggs need no loot table: death already drops everything they carried and held.
- **Weapon attacks:** pressing `F` with a main-hand weapon equipped swings along the trogg's exact aim vector and damages the nearest online, living trogg or dark creature — **a dark creature outranks a trogg at equal distance** (so a swing meant for a hostile creature doesn't clip a bystanding trogg first, now that the two are meant to be fighting side by side rather than each other) — whose **hit circle** is in reach: within `MELEE_RANGE_TILES` (1.7 *(initial)*, centre to circle edge) and inside the ±55° swing cone (`MELEE_ARC_RAD`; point-blank inside `MELEE_POINT_BLANK_TILES` waives the angle, and a target's radius widens the cone by what its silhouette subtends — `shared/combat.ts`). Troggs have a 0.45-tile radius; dark creatures are scaled per species, so a larger one is as hittable as it looks. The client passes its aim, but the server re-derives every position with the same projections the client renders (invariant 3).
- **Bare fists always work** (`UNARMED_DAMAGE`, 5–10 *(initial)*): an empty-handed `F` swings too — the weakest weapon, stored as the `"fists"` action impulse so every client animates the bare swing. Fists follow the same swing resolution as weapons, off-tool node scratches included.
- **Out-of-combat regeneration**: a creature untouched for `HEALTH_REGEN_DELAY_MS` (30s *(initial)*) heals `HEALTH_REGEN_FRACTION` (5%) of its max, rounded up, every `HEALTH_REGEN_TICK_MS` (3s) — the scheduled `regenCreatures` sweep, the same sanctioned timer exception as the ember/dark-creature wander. `lastDamagedAt` on the player and dark-creature rows is the out-of-combat clock. Dead troggs never regen — respawn restores them.
- **Every weapon rolls its own damage range** (`WEAPON_DAMAGE` *(initial)*: sword 20–30, axe 14–22, pickaxe 11–19, shovel 8–16, inclusive). An accepted hit rolls inside the weapon's range with the reducer's context RNG, so no two swings land identically while replay stays deterministic. Gathering tools resolve their node first — a pickaxe swing that reaches a boulder mines it and spares the creature behind it; an axe fells the tree — and wound creatures only when no node claims the swing. Off-hand items (shield, torch) and bare fists deal nothing.
- **A shield blocks incoming damage** (`SHIELD_BLOCK_FRACTION`, 30% *(initial)*): a trogg with a shield equipped in the off hand takes that fraction less on every hit, melee or thrown, rounded to the nearest point (`damagePlayer`, `shared/constants.ts`). It doesn't reduce the invulnerability cheat's zero damage, and dark creatures — which never equip — always take a hit's full amount.
- **Floating damage numbers:** every health drop a client sees (weapon hits and thrown objects alike) pops a rising, fading damage number over the target — anchored to the world, not the target, so a killing blow's number outlives its victim. Client-side display only, derived from synced health deltas.
- **Thrown objects:** pressing `F` while carrying throws the held object along the **exact aim** (free-direction, not the four cardinals; `throwCarried` marches the aim ray tile by tile) up to `THROWN_OBJECT_RANGE` (4 tiles *(initial)*). It damages the first online, living trogg or dark creature it hits by `THROWN_OBJECT_DAMAGE` (40 *(initial)*) and then lands near the impact; if it hits no damageable character, it lands on the last clear tile the ray reaches before range or a blocker. This is still an input-driven reducer, not a projectile simulation — the server resolves the landing at once. The **client animates the flight**: the object arcs (a tumbling parabola) from the thrower's hands to the landing spot, and the real row is held out of the scene until it touches down (`world.ts`, cosmetic). The mechanic is currently dormant in practice: nothing is carryable at this design's outset (see "Interacting"), so there's nothing to throw — this stays engine capability without a live use case until a new carryable calls for it.
- **Use cooldown:** the server drops an equipment use that lands inside the previous swing (`EQUIPMENT_USE_COOLDOWN_MS`, 250 ms *(initial)*, just under the 300 ms `EQUIPMENT_ACTION_MS` impulse), so attack rate is capped by the animation cadence, not by how fast a client can call the reducer (invariant 3).
- **No twitch checks:** combat has no projectiles, physics, cursor aiming, or per-frame attack checks — a use is one input-driven reducer resolving static hit circles at that instant.
- **Death:** when damage takes a trogg to zero health, the server marks it `dead`, stops its motion on its current tile, and leaves it online so other players can see it — lying flat in the dead stance (every creature shares it), faded translucent, under its respawn countdown. Dead troggs cannot move, interact, spawn objects, or use equipment. If a dying trogg is carrying a tile-sized object, the server tries to drop it at the death tile so nothing is orphaned. Every inventory row, including equipped items, is removed from the trogg and dropped as `ground_item` rows nearby; stack quantities are preserved.
- **Respawn:** death stamps `respawnAt` five seconds in the future (`PLAYER_RESPAWN_MS`, 5000 *(initial)*) and inserts a one-shot `player_respawn` scheduled row. Dead troggs show a visible respawn countdown. When the timer fires, the same `player` row returns at full health with the same name, colour, and style, **just outside your cave** — the world's coast alcove (`EMERGE_ARRIVAL`), where you first emerged and where you descend back down. A trogg that died inside its own birth cave is pulled out to the alcove too (the cave is for births, not a spawn room), which the client sees as a zone transfer.

### The fire and the dark

The game's central system: the tribe holds back an encroaching dark by keeping fires lit, and everything else — gathering, combat, presence, even world generation — answers to whether the fire is fed. This section is the load-bearing one; when it and any other section disagree, this one wins.

#### The stockpile

- **One shared pool, no personal share.** The tribe's raw resources — Stone, Wood, Glowcap Mushroom, and whatever else gathering yields — live in one global `stockpile` table (item, qty), fed directly by every gather action, bright or ember (see "Gathering", "Inventory"). Nobody owns a portion of it, nobody withdraws it into a personal stack; it exists only to be spent on brazier upkeep (below). It is read-only from a player's perspective — visible to everyone, mutated only by the reducers that gather into it and the ones that burn it.
- **Capped, not hoardable.** The stockpile holds at most `STOCKPILE_CAP` *(initial)*, sized to a few days of the tribe's current total upkeep. A full pool doesn't grow further — gathering past the cap is wasted effort, communicated the same way a full inventory is. The cap exists so a long-idle tribe can't bank enough surplus to stall the dark indefinitely (see Territory and permanence, below): the world's fate always answers to recent presence, not historical accumulation.
- **Per-player contribution tracking is an open thread** (see Open design threads): a counter of what each trogg has put into the stockpile, for dashboards and bragging rights, without anyone being able to take out more than they put in — because nobody takes out at all.

#### Presence: bright, ember, dormant

- **Bright** is the state of a trogg whose player is connected and playing: full judgment, full combat, full access to the frontline and the dark beyond it. Everything a trogg can do today, it does at this state.
- **Ember** begins the moment a player disconnects, provided the trogg has `kindlingCharge` remaining. An ember trogg does not leave the world (it isn't dropped from view the way a plain disconnect works today) — it keeps working, driven by the same scheduled-reducer wander pattern that moved the retired Hogs: ambling between safe interior nodes, gathering on instinct, depositing into the stockpile at `EMBER_EFFICIENCY_FRACTION` *(initial)*, roughly 25–35%, of a bright trogg's rate. An ember trogg earns no XP (Skills and XP) — instinct isn't judgment. It cannot leave lit territory, cannot fight, and — because dark creatures can't cross into light (Dark creatures) — is safe by construction rather than by a special-cased invulnerability flag.
- **Dormant** begins when kindling charge reaches zero. The trogg settles at the nearest hearth, stops moving and stops contributing, and simply waits — present, visible, a quiet fixture proving someone once played here, exactly the texture that keeps the world from reading empty even at the game's real scale of a handful of concurrent players. Reconnecting returns a trogg to bright instantly, wherever it settled, the same identity-resumes-on-reconnect guarantee the game already gives every player (Identity).
- **Kindling charge** is what a trogg earns for going ember at all, and it's the whole answer to "why can't someone just script a thousand guests to graze forever": charge accrues only from genuine bright play — real input over real time (invariant 2's grammar, not a connected-but-idle socket) — at `CHARGE_ACCRUAL_RATE` *(initial)* per minute played, capped at `CHARGE_MAX` *(initial)*, and decays at `CHARGE_DECAY_RATE` *(initial)* per hour once ember. A trogg nobody actually plays runs dry and goes dormant quickly; keeping a trogg productive costs the same thing it always should — showing up. This is also literally what design pillar 6 is describing.
- **Recall on disconnect from unclaimed ground.** A bright trogg that disconnects while out past the frontline or in the penumbra doesn't get to linger there — it's recalled to the nearest hearth before going ember, the same `nearestSafeTile`-style rescue the game already does for a trogg stranded by a world regen (Zones). Ember troggs only ever exist on lit, safe ground.

#### Territory and permanence

- **A region is the unit of safety, not a radius.** Every world region (see "region" in the glossary) is either **interior** with its own brazier lit, or it isn't — there is no partial safety within a region's borders. A dark creature cannot enter any tile of a region whose brazier is lit (Dark creatures); the union of every such region is the tribe's claimed territory. A brazier's rendered glow size is cosmetic, not a gameplay boundary.
- **Explored and safe are different facts, and only one of them is permanent.** Claiming a region makes it explored forever: it always renders, always collides as real ground, and only the debug "Reset frontier" tool ever forgets it (Generation). Whether it's currently *safe* is a separate, revocable fact — whether its one brazier is lit — that upkeep (below) can take away without touching the first.
- **Upkeep scales with how much ground you're holding, not its shape.** Each lit, non-eternal brazier draws a steady trickle from the stockpile (`BRAZIER_UPKEEP_RATE` *(initial)*, per brazier per unit time) for as long as it burns — one brazier per claimed region, so total upkeep grows with the number of regions claimed rather than a frontier's perimeter length.
- **Unpaid upkeep recedes the frontline, deepest region first.** When the stockpile can't cover total upkeep, the brazier in the region furthest from the Hearth by claim-graph hop-depth (not raw tile distance — regions vary too much in size and shape for that to mean the same thing everywhere) gutters and goes dark, in order — never at random, never a shallower region while a deeper one still burns. Guttering only turns a region unsafe again: its dark creatures return and it starts replenishing them (Dark creatures → Territory-linked respawn), but the region itself stays exactly as explored and walkable as it was. This is weather, not loss (design pillar 6's flip side: presence sustains as directly as its absence recedes).
- **The First Fire never gutters.** Whatever else recedes, the origin hearth at the Hearth is eternal — however long the world goes untended, a returning player always lands somewhere lit, warm, and playable. The game has no dead state, only a floor.
- **Permanence follows the ground, not the kill.** A dark creature killed on ground the dark still owns respawns — the dark replenishes itself, and idle skirmishing at the edge can't grind the frontier down by attrition alone. A dark creature killed on lit, claimed ground stays dead, because nothing dark can persist where a hearth already holds the light. Claiming a region (below) is therefore the only way to make a kill there permanent.

#### Territory claiming

Pushing the frontline outward is earned by clearing ground, not a separate event bolted on top of it:

- **Kill everything in a region, then set a brazier down — free.** A penumbra region's own living dark creatures are the only thing standing between it and a hearth. Once every one of them is dead (a corpse doesn't count), standing anywhere in that region and pressing empty-handed `E` sets a brazier down on the spot — or the nearest free tile — and claims the region: no component, no fuel, no timer. The cost is paid in the fight, not at the fire.
- **The corpse-respawn timer is the actual pressure.** A dark creature's corpse reaps after `NPC_CORPSE_MS` and respawns a fresh one if the ground it died on is still unsafe at that moment (Dark creatures → Territory-linked respawn) — which an unclaimed penumbra region always is. A group has to out-pace that clock across the region's *whole* population, together, or the dark simply refills behind them.
- **A claim also exposes the next layer of the frontier.** The same reducer call that claims a region moves it from penumbra to interior and seeds its own still-unclaimed neighbours as the frontier's new penumbra, populated immediately (Generation) — so the next scout already has somewhere fresh to find, without waiting on anyone else's claim first.
- **Relighting a guttered brazier is always free, anywhere already interior.** The region never stopped being claimed while its brazier was dark, only unsafe (Territory and permanence), so putting it back doesn't require re-clearing anything: empty-handed `E` beside a guttered, non-eternal brazier relights it on the spot, whatever's alive nearby.

#### Generation: only as far as the light reaches

- **The map has no edge.** Region capitals aren't a hand-authored list — they resolve on demand from an unbounded, deterministic lattice over the plane: divide it into fixed-size cells (`REGION_LATTICE_CELL`, 70 tiles *(initial)* — roughly today's average region scale), and every cell yields exactly one capital, its position jittered within the cell (up to `REGION_JITTER_FRACTION`, 40% of the cell *(initial)*, so it can't drift into a neighbouring cell's share of the plane) by a hash of `(cellX, cellY, WORLD_SEED)`. The same hash assigns a biome (drawn from the `BiomeParams`/`biomePalette` catalog) and a generated name (a two-part word-bank combiner tuned to the game's naming flavor — "Rust Gallery," "Emberrift," "Boneyard"). `regionAt(x, y)` resolves any tile, however far out, by checking candidate capitals in its own lattice cell and the 3×3 neighbourhood of cells around it — sufficient because `REGION_JITTER_FRACTION` caps how far a capital can drift from its cell's centre, so nothing further out could ever be nearest; resolving one tile never needs to know about capitals anywhere else on the plane. Deterministic and shared (invariant 7): client and server compute the identical region for the identical tile with zero network round-trip.
- **The Hearth is the one hard-coded region.** The origin lattice cell is always "The Hearth," always biome `cave`, and is where the First Fire, the starting town, the spawn point, and the birth-cave mouth live — the game's literal starting point, not a generic biome roll like everywhere else.
- **Tiles synthesize on demand, chunk by chunk** — matching how the client already streams geometry around the camera (Zones). Cave/rock fill probability comes from a tile's region's biome, smoothed over its 3×3 neighbourhood (already local, unaffected by unbounded coordinates). Reachability is guaranteed by construction rather than checked after the fact: the moment a region's capital resolves, a corridor — a biased random walk, the same technique the Hearth's own rivers use — is carved connecting it to **every** Voronoi-adjacent neighbour's capital (a full mesh, not just enough for one path back to the Hearth), so any two regions that touch on the map are always directly walkable between. Boulders, trees, and dark creatures seed per region from an RNG stream keyed off that region's own capital coordinates, at a density that scales with the region's claim-graph hop-depth from the Hearth up to `MAX_DEPTH_DENSITY_MULTIPLIER` (4× the Hearth-adjacent baseline *(initial)*) — deeper regions are tougher to clear, not just further away (Territory claiming), but never past a ceiling a group could never actually clear.
- **A modest chance of local water.** Beyond the Hearth's own two rivers, each region rolls a `REGION_WATER_CHANCE` (30% *(initial)*) chance, from the same per-region hash, of a small pond or short stream near its capital — a few tiles of shallow or deep water, not a carved river crossing the whole region. Purely cosmetic variety; claiming, safety, and reachability never depend on it.
- **The reveal unit is a region, not a ring.** Every region is **interior** (claimed — a brazier's been lit there), **penumbra** (known, adjacent to an interior region, not yet claimed — walkable, dangerous, scoutable, populated), or **unreached** (no `revealed_region` row at all yet). Unlocking a whole named biome at a time, rather than the light creeping outward in every direction the same distance, means logging in can genuinely surface a new place to go, and other players' claims open ground you haven't seen yet — not just a slightly wider circle around the same Hearth.
- **A region's name is locked the first time it's scouted, not recomputed on demand.** Position, biome, and tiles stay pure functions of coordinates alone — but a display name has to be unique among every currently-known region, which one region's own coordinates can't guarantee by themselves. The moment a region is first exposed as penumbra, the server resolves its hash-derived candidate name, checks it against every other row already in `revealed_region` (interior or penumbra — the table is still small), and rerolls (a deterministic secondary hash) on collision until it's unique, then inserts the row. Claiming a region later only flips that same row's `interior` flag — the name was already locked the moment a scout could see it, and never changes again.
- **Penumbra and unreached both render — neither is a wall.** A penumbra region renders and collides as its real, generated terrain — walkable, dangerous, exactly like an interior region except unsafe and unclaimed — under light fog (client-only, no gameplay effect: a haze wash on the ground plus slowly drifting mist banks, so unclaimed ground reads as weather, not frost) that reads as scouted-but-untamed. An unreached region renders its real terrain too, under a much heavier fog, but stays a hard collision wall regardless — fogged, never solid rock, so what lies beyond always reads as unclear rather than nonexistent. A scout only ever sees clearly as far as one claimed region out; past that the fog just thickens, with no far edge to eventually reach, because there genuinely isn't one.
- **Crossing into penumbra says so.** There's no gate or wall at the frontier — regions are a soft Voronoi line in continuous, open terrain — so the fog tint alone is easy to miss mid-exploration. The client raises a one-line banner ("Entering \<Region\> — unclaimed") the moment your own tile's region changes to one the tribe hasn't claimed; it fires once per region entered, purely client-side, no gameplay effect.
- **The Hearth starts interior**, its neighbours seeded as the initial penumbra the moment the world exists (on first connect) — so there's already somewhere to scout on day one, without anyone claiming anything first. The moment a region becomes penumbra — whether at world start or because a fresh claim just exposed its neighbour — its name locks and its dark-creature seeds generate and go live immediately (Territory claiming), so a scout finds real stakes there before anyone's claimed it.
- **Claiming is exactly the mechanic described above** (Territory claiming) — the only thing region-scale adds is where a claim's neighbours come from: `neighborsOf(slug)` runs the same bounded lattice-neighbour search instead of a lookup table, and it flips an existing row's `interior` flag rather than inserting a fresh one, since the region already has a row from the moment it first became penumbra.
- **Revealed ground is frozen.** Claiming a region is one-way in ordinary play (only the debug "Reset frontier" tool undoes it), and a region's capital, biome, name, and tiles never change once resolved — the lattice hash behind them, and the locked name, are both fixed, so nothing regenerates out from under a player.
- **Nothing durable grows except region identity, and it grows slowly.** Position, biome, tiles, and every seed stay pure functions of a tile's coordinates and the world seed — never stored, never synced over the network. Only a region's locked name and claim status live in a database row (`revealed_region`, one row per region ever scouted, not only claimed ones — see Data model), and that table still grows by roughly one row per claim plus its handful of newly-exposed neighbours, nowhere near the size of the map itself. Terrain and seeds never touch the network at all; only this small, named-and-claimed layer on top of them does.
- **A trogg's own claim-depth is fine to show; the world's total size never is.** A UI stat like "N regions deep" or a similar per-tribe progress readout doesn't leak how big the world actually is, since there's no known ceiling to compare it against — it's a personal-progress number, not a spoiler. What must never appear anywhere player-facing is anything that implies a *total* (a region count that reads as "out of," a leaderboard against a known maximum).
- **This is also the cosmology.** The world doesn't hide beyond the light — it doesn't yet exist there, and there is no point at which that stops being true.

### Onboarding: the Warren

Every trogg is born in the dark — **alone**. Onboarding is single-player by construction: each newborn gets a private copy of the shared `birthcave` template (**The Old Dark**, `generateBirthCave` — a small, fully enclosed cave), scoped by its own zone id `birth:<identity hex>` (`BIRTH_ZONE_PREFIX`). Zone-filtered subscriptions mean nobody else ever sees or reaches into it, so nothing can be stolen (the pickaxe grief is impossible) and no occupancy/reclaim machinery exists at all.

The arc: wake in the **middle of the glowmoss cavern** (a hearth ember lights the wake-up spot; the opening camera is raised and close), pick up the pickaxe beside you (`E`), equip (`I`), mine through the **rubble plug** — two rows spanning the cavern end of a long throat (`F`) — gather the stones the rocks drop, and **walk the throat out — the walk is the door** (the transfer threshold sits at its far end, so nothing whisks you away before you've picked up your haul). The throat **tapers as it climbs** — three wide leaving the cavern, then two, then one at the exit — and the cave never shows the outside: **glowmoss pools spaced along the walk** (template-stamped; the client lights each one, the sealed exit landing brightest) lead toward the end without ever revealing it. The `emerge` reducer verifies proximity server-side (invariant 3) and transfers the trogg to `world` at `EMERGE_ARRIVAL`, a cave-mouth alcove carved into the south coast. The client reboots into the new zone behind the boot screen; the arrival opens on a raised reveal shot, fires `warren_emerged`, and plays the **first dawn** (~12s local-only blend from morning gold into the shared sky; the shared sky lock outranks it).

- **Rubble is the boulder mechanic verbatim** — rows in the newborn's zone id; mining, hit feedback, and the stone drop just work. World-zone tools (`resetBoulders`, the registry seeder, the spawn cap) never touch other zones' rows.
- **The cave is yours and it persists**: nothing is wiped on emerge — your mined-out neck stays mined. In the world, the arrival alcove's deep end is a dark tunnel mouth; pushing into it (`enterCave`, same client-detect/server-verify shape) descends into your own cave, landing below the neck so arriving never bounces straight back out. Every trogg keeps its cave; nobody else's is reachable.
- The client boots the zone named on the player row (a newborn's cave or the world); `getZone` resolves any `birth:*` id to the shared template. Terrain palettes use regions only on the world map — other zones are one biome throughout.
- Deaths respawn just outside your cave (the coast alcove, `EMERGE_ARRIVAL`), including a newborn that dies mid-dig — the cave is for births, not a spawn room.
- Open threads: the emergence sightline (the First Fire visible from the coast, walk-to-Hearth breadcrumbs, naming at the fire).

### Debug cheats (Commands drawer)

Developer/alpha-tester tools in the Commands drawer (a lone top-right toggle — labelled **Debug Tools** — opening a right-edge drawer; the toggle rides the drawer's edge so the same button closes it), gated by the `cheat-commands` flag. They exist both for debugging and so alpha testers can feel the game out and escape any weird spot they lock themselves into. Cheats live on the player row (`cheatSpeed`/`cheatFly`/`cheatNoclip`/`cheatInvulnerable`, plus the flight motion columns `z`/`dirZ`) and are written only by dedicated reducers that **settle motion first** (a rules change must never replay elapsed motion history at new rules) and clamp every value (invariant 3).

- **Speed ×3** (`CHEAT_SPEED_MULTIPLIER`, 3 *(initial)*): multiplies the shared projection's tiles-per-second; rides the row like `running`, so every client derives the same faster motion.
- **Fly:** airborne, with real synced altitude. `z` is a linear derivation origin like `x`/`y` and `dirZ` the vertical intent, written on **Space/C input transitions** (`setLift` — the `move` pattern's third axis, invariant 2): Space climbs, C sinks, release holds, clamped to `FLY_MAX_HEIGHT` (14 *(initial)*) at `FLY_VERTICAL_TILES_PER_SEC` (5 *(initial)*). Horizontal collision is **height-aware**: a flyer clears whatever sits below it — deep water above `FLY_CLEAR_WATER` (0.2), trees/boulders/creatures above `FLY_CLEAR_OBSTACLE` (3), rock walls above `FLY_CLEAR_ROCK` (11) — judged at the motion's origin altitude, which every input transition and per-tile rebase refreshes. Every client renders a flyer at its true derived altitude.
- **Noclip:** the grounded clearance — walk through anything; the projection keeps only the zone rectangle.
- Switching fly or noclip **off** while inside geometry settles the trogg to the nearest standable tile; landing zeroes `z`/`dirZ`.
- **God mode:** `damagePlayer` no-ops on the target; the attacker's swing still lands and animates.
- **Heal** (`healSelf`): full health for a living trogg — respawn already handles the dead.
- **Unstuck** (`rescue`): settle, ground, and step to the nearest standable tile (spawn as the last resort) — the universal escape hatch.
- **Sky lock (World section):** a slider pins the day-night phase for **everyone** — the sky is shared fiction, so a scrubbed noon is noon on every client. The override lives in the public `world_state` singleton (id 0, `skyLocked`/`skyPhase`), written by `setSky` (phase wrapped into [0, 1)); "Live" releases it back to the shared wall clock. Clients subscribe to the row and read it in their daylight pass; the drawer's controls paint from it, so every open drawer agrees.
- **Reveal next region:** claims one currently-penumbra region directly, skipping the usual clear-the-zone requirement, and inserts a lit brazier there so the shortcut leaves the same end state a real claim would (see "The fire and the dark" → Generation).
- **Jump N regions out:** claims a chain of N regions directly outward from the current frontier in one shot, the same shortcut as "Reveal next region" repeated N times — for testing generation at genuine distance without manually claiming hundreds of regions in sequence. Dev/testing only.
- **Reset frontier:** clears every claimed region back to just the Hearth. Entities already seeded in now-unclaimed regions are left in place — simply unreachable behind the reveal gate again.

### Crafting

- Recipes: inputs → output, optional skill/level requirement, crafted at stations at the Hearth (e.g., torches, better tools). Bulk inputs draw from the shared stockpile, not personal stacks — consistent with gathering's direct-deposit model (see "The fire and the dark" → The stockpile); nobody hoards materials personally, crafting included.
- Better tools gather faster or unlock harder node types — the progression spine.

## Data model

One layer. **SpacetimeDB** is the durable store *and* the live feed: the tables below are the source of truth, and clients subscribe directly to the rows in their current zone — there is no separate cache or room projection to keep in sync. Only trusted module entrypoints may write: reducers for pure game mutations and procedure wrappers that open a SpacetimeDB transaction with `ctx.withTx(...)` before doing external telemetry. The writer's identity is the connection's own `ctx.sender` (invariant 3). A connecting client upserts its `player` row (`clientConnected`) and settles it on disconnect (`clientDisconnected`); SpacetimeDB persists every table, so a returning Identity resumes its trogg with no hydrate step. Motion intents (direction, `movedAt`) live in the durable row, but position is still *derived* from them with `projectMotion`, never advanced on a timer — the no-teleport-by-quit rule holds (invariant 1). A zone with no connected players produces no diffs and no work. Indexes noted where the access pattern demands them.

Dev mirrors prod: a local `spacetime start` instance runs the very module production runs — `just dev` publishes to it and regenerates the client bindings — so persistence is exercised the same way it runs in production. No Docker, no separate database to provision.

## Multiplayer scaling stance

- **One subscription per zone, naive within it.** A client subscribes to the rows of its current zone; SpacetimeDB pushes row diffs to everyone subscribed. Within a zone there's no interest management and no instancing — everyone sees every player.
- **Fixed cost, watched in metrics.** The VPS runs the self-hosted SpacetimeDB instance on a flat monthly bill, not a usage meter, so the concern is CPU, memory, and bandwidth — watched via server metrics and PostHog, not an egress invoice. At the realistic scale (tens of concurrent) a single instance on a small Hetzner box is ample.
- **Capacity cap before launch.** Connection and subscription limits are set so a viral night sheds or queues load instead of toppling the box; vertical scale (a bigger VPS) is the first lever. Per-zone entity caps (`MAX_DARK_CREATURES_PER_ZONE`, `MAX_BOULDERS_PER_ZONE`, `MAX_GROUND_ITEMS_PER_ZONE`) bound Commands panel spawns and carried-object drops server-side, so no client can flood a zone and inflate every wander tick.
- **The answer key, only when a graph demands it:** narrow the subscription with area-of-interest SQL filters (a smaller `WHERE` per client), crowd aggregation above a density threshold, and — when one box isn't enough — SpacetimeDB's own horizontal scaling. None of these are built in advance.
- **Same module, dev to prod.** The instance the VPS runs is the module that runs locally, mirrored from the current production module. Running beyond a single instance — and everything else in the answer key — stays deferred under invariant 10.
- **Swappable position feed:** the client reads positions through the zone subscription and writes through reducers, both in one place. Tightening to area-of-interest is a query change, not a rewrite.

## Invariants (non-negotiable)

1. No simulation tick. We never run a server simulation loop; state changes only inside a reducer — on player input or a scheduled reducer (SpacetimeDB's deterministic timer, used for respawns, action completions, and the ember/dark-creature wander). SpacetimeDB sends only row diffs to subscribers, so an unchanged table produces none and a zone with no connected players computes nothing.
2. No per-frame or per-tile server sync. Movement intents (click→path, key down/up, direction change) are sent via reducers; clients derive and predict motion locally. Synced player state changes on input, never on a timer or every frame.
3. All authoritative state lives on the server — SpacetimeDB's durable tables, written only by reducers or procedure transactions, with the writer identified by `ctx.sender`. Never trust the client.
4. Analytics events are low-volume and never contain chat content or PII beyond the player name (full rules in [analytics.md](analytics.md)).
5. Feature flags are operational controls, not a blanket requirement. Add or configure one when a feature needs remote rollout, a kill-switch, an experiment, or live tuning; otherwise keep the code simple. Any flag key the code reads must be registered in [analytics.md](analytics.md) with its fallback behavior, and the matching PostHog flag must be created or updated in the configured project during the same task.
6. The game is playable at the end of every session. No half-wired states on `main`.
7. No twitch combat: combat is tile-based, stat-driven, and slow — no projectiles, physics, real-time aiming, or per-frame attack checks. Generation is deterministic and shared: every tile is a pure function of its coordinates and the world seed, computed identically by client and server — never a live or per-client random source; which of it is reachable is a separate, durable reveal state, not regenerated terrain (see "The fire and the dark" → Generation).
8. No secrets in the repo — env vars only, `.env*` is gitignored. This repo is public.
9. Glossary names are canonical across code, schema, events, and UI.
10. No preemptive scaling work. Optimizations from the scaling answer key are built only when a dashboard graph justifies them. No instancing below ~1,000 concurrent in a zone.

## Roadmap and current state

Roadmap notes are planning context, not permission gates. Pick work by current product need, maintainer direction, and what keeps the game playable. Do not block a task because older notes placed it later, and do not treat this section as a release checklist.

**The shipped code implements this spec.** The fire-and-dark migration that pivoted the game away from the Hog-town/personal-inventory design is complete, in the dependency order it was planned:

1. **Retire Hogs** — the `hog` table, the roaming-Hogs wander reducer, Hog-specific carry/throw/ball-form rendering, the merchant/currency threads, and the `roaming-hogs`/`hog-reset` flags are gone; the old hub zone's town dressing lives at the Hearth inside the one seamless `world` zone.
2. **Stockpile and direct-deposit gathering** — the global `stockpile` table exists and mining/woodcutting deposit into it directly; bulk raw resources never enter personal inventory.
3. **Hearths, braziers, and territory** — the `brazier` table with the First Fire as the one eternal row, region-wide "dark creatures can't enter lit ground" collision, upkeep drain off the stockpile, and guttering ordered by claim-graph hop-depth (deepest claimed region first).
4. **Presence: bright / ember / dormant** — `kindlingCharge` and its anchor on the player row, ember troggs that keep working interior ground after a disconnect, dormancy at zero charge, and the recall-to-hearth for a disconnect on unclaimed ground.
5. **Dark creatures** — the grask, on the shared rig/motion machinery: aggro-on-sight, light-bound movement, territory-linked respawn.
6. **Territory claiming** — empty-handed `E` in a cleared penumbra region sets a brazier down for free and claims it; relighting a guttered brazier in interior ground is likewise free; the corpse-respawn clock is the claim pressure.
7. **Infinite, region-at-a-time worldgen** — the unbounded capital lattice (`shared/worldgen.ts`): on-demand chunk synthesis, full-mesh corridors between lattice-adjacent capitals, depth-scaled seed density, `revealed_region` rows (`slug, name, interior, revealedAt`) with names locked at first scouting, fog-rendered penumbra/unreached ground, the unclaimed-crossing banner, and the frontier debug tools (reveal next, jump N out, reset).

Flag any place code and doc disagree rather than silently trusting either (per "How to use this document").

Implemented world systems that carried over unchanged through the pivot: zone-scoped subscriptions, per-tile walkability, free camera-relative WASD movement with sliding collision, boulder mining and tree felling, the jointed 3D rig and its animation/equip system, inventory/equipment mechanics for personal gear, sword/unarmed combat grammar (hit circles, swing cones, weapon damage, shield block), trogg health/death/respawn, hold-shift-to-run, recolouring/restyling, chat, the ghost easter egg, the Commands drawer, and anonymous-first identity with optional Discord claim.

Likely next work areas beyond the migration list above: naming the first dark-creature species and any Umbra-touched frontline variants, the exact claim-pacing design, crafting recipe specifics at Hearth stations, and how a respawning or newborn trogg might eventually choose which front to arrive near. These are intentionally fluid; implement the slice that best serves the current task.

## Open design threads

- Dark-creature bestiary: species names, stats, loot tables, and Umbra-touched frontline variants of the same base models — the model/rig work reuses everything Hogs proved out (Avatars and equipment), so this is content, not engine work.
- Fronts: whether the frontline eventually forks into distinct directions/biomes the tribe can choose between, each with its own hazards and materials feeding the shared stockpile — floated in design discussion, not committed.
- Migrating spawn: whether a newborn or respawning trogg should eventually surface near the tribe's current frontier rather than always at the Hearth, so new and returning players inherit whatever ground has already been won.
- Crafting station specifics at the Hearth: which recipes exist, skill/level gates, and station placement/UI (sourcing from the stockpile is decided — see Crafting).
- Zone and terrain art direction once the plain cave/interior stops being charming.
- Own-avatar prediction polish: basic optimistic acknowledgement handling has landed for movement; richer rollback/smoothing is deferred until playtests show a need.
- Region-grid contiguity: each named region should always be a single connected area, with no stray tile of one region embedded inside another's claimed ground (which would otherwise sit unclaimed and unlit long after its surroundings are safe). The bounded, per-region lattice-neighbour generation (Generation, above) shouldn't produce this class of bug the way a single whole-map domain-warp pass could, but it's worth an explicit invariant test regardless rather than assuming it by construction.

Cut with this pivot, and not carried forward: currency design and Hog/mascot integration (retired with the Hog-town design); light/darkness as a mechanic (no longer an open thread — it's the spine of the game now, specced in "The fire and the dark").
